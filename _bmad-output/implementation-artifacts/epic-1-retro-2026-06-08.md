## 어려웠던 것 / 개선점

### 1. Python 생태계 함정이 8개 중 5개 스토리에서 발생
런타임에서만 드러나는 함정들이 반복됐다:
- `alembic.ini` 한글 주석 금지 (configparser cp949)
- `pydantic-settings` v2 list 파싱 크래시 (JSON 디코드 시도)
- `uuid_extensions` import 경로 (`import uuid7` ≠ `from uuid_extensions import uuid7`)
- `setup-uv` v6 → v8.1.0 버전 핀 (moving tag 미발행 정책 변경)
- `python-preference=only-system` 미적용 시 'no system python'

deferred-work.md에 기록은 됐으나, story 파일의 `⚠️ 알려진 함정` 섹션이 이를 예방하는 역할을 하고 있다. Epic 2에서도 동일하게 유지한다.

### 2. 보안 체크가 dev-story 단계에서 명시적으로 없었다
구현 단계에서 'AC 충족'이었던 보안 버그를 코드 리뷰에서 잡은 사례가 4건. 코드 리뷰가 안전망 역할을 했지만, dev-story 단계에서도 인증/인가 관련 코드 작성 시 아래 항목을 의식적으로 체크하면 review 사이클을 줄일 수 있다:
- 공개/보호 엔드포인트 401 분기 처리
- 입력값 상한 (password max_length, 공백 처리)
- 타이밍 분기 (Argon2 더미 해시 패턴)
- payload 파싱 예외 전체 catch (KeyError/ValueError/TypeError/AttributeError)

### 3. 소프트삭제 + Unique 인덱스 패턴이 두 번 반복됐다
1.3(이메일)과 1.6(카테고리명) 모두 동일한 패턴: `get_by_*`는 `deleted_at IS NULL` 필터, 인덱스는 전역 unique → 소프트삭제 행이 잔존하면 재삽입 시 IntegrityError. 두 건 모두 "현재 삭제 경로 없어 도달 불가"로 defer. **Epic 6 이전에 `partial unique index(WHERE deleted_at IS NULL)`로 일괄 처리 검토.**

### 4. NO_VCS 상태로 Epic 1 전체 진행
8개 스토리 동안 git 저장소 없이 개발 → CI 검증이 로컬에서 불가능한 상태로 1.8 산출물 작성. 결과적으로 push 후 Node 20→22 패치가 한 번 필요했다. Epic 2부터는 git 저장소가 있으므로 CI 그린 유지가 DoD의 일부가 된다.
