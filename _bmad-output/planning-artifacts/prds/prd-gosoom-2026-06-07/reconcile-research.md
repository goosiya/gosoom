# 리서치 정합성 검토 보고서

원본 기술 리서치(`technical-gosoom-mvp-tech-stack-research-2026-06-07.md`)의 핵심 결정·제약·리스크가
완성된 PRD 본문과 addendum에 충실히 보존되었는지 대조했다. PRD는 의도적으로 기술 결정을 본문에서 빼고
addendum에 보존하는 구조이므로, 검토 초점은 ① addendum이 리서치의 결정/제약/리스크를 충실히 옮겼는가,
② 제품에 영향을 주는 제약이 PRD의 NFR에 누락 없이 표면화되었는가 두 가지다.

---

## 종합 판정

- **"5대 핵심 기술 결정"은 전부 충실히 이관됨** (양호). addendum "5대 핵심 기술 결정" 표(행 1~5)가
  리서치 종합 섹션의 5대 결정(리서치 343~347행)과 1:1로 정확히 대응한다. 채팅 폴링→NFR9/CM1,
  자동 일시정지→NFR8, refresh 토큰→FR3, 앱 레벨 권한→NFR4, FastAPI 단일 소유 전제(SQLAlchemy+Alembic)→NFR6
  까지 PRD 측에도 잘 반영되어 있다.
- **누락/축소(distortion)는 주로 두 블록에 집중**: ① 리서치 리스크 표(303~309행)의 일부 리스크·완화책,
  ② 주제 3(이관, 104~111행)의 마이그레이션 도구·절차 디테일.
- **올바르게 생략된 항목(가점 아님, 확인용):** Supabase Realtime "동시 연결 200개" 제한(리서치 124행)은
  addendum에 없는 것이 **정상**이다. 폴링 채택으로 realtime을 쓰지 않으므로 무관하다.

---

## 발견된 갭 목록 (심각도 순)

### GAP-1. 자유 티어 하드 한도(특히 DB 500MB)가 어디에도 표면화되지 않음 — 심각도: 중

- **리서치 근거:** 124행 — "DB 500MB, 파일 1GB, MAU 5만, realtime 동시연결 200, 프로젝트 2개."
- **현황:** addendum "운영 주의"는 **자동 일시정지만** 보존하고 용량 한도를 전부 누락했다.
  PRD에도 데이터 용량/보존 한도에 대한 NFR이 없다.
- **제품 영향(중요):** 이 갭은 단순 누락이 아니라 **PRD 내부 정책과 충돌**한다.
  - PRD **FR19 / R3**: 비활성화된 계정의 기존 요청·채팅 데이터를 **삭제하지 않고 무기한 유지**한다.
  - PRD **FR23 / 채팅(FR15~18)**: 채팅 메시지는 보존되며 관리자가 열람한다.
  - → 즉 데이터는 **단조 증가**하는데 Phase 1 호스팅(Supabase 무료)은 **500MB 상한**을 갖는다.
    시연/초기 단계에선 여유가 있으나, 이 제약이 NFR에 전혀 등장하지 않아 운영·이관 판단의 근거가 사라졌다.
- **권고:** addendum "운영 주의"에 무료 티어 한도(DB 500MB·MAU 5만·프로젝트 2개)를 복원하고,
  PRD NFR(예: NFR7 데이터 무결성 또는 신규 용량 NFR)에 "Phase 1 저장 용량 상한 존재 → 한도 근접 시
  Pro/Railway 전환" 트리거를 한 줄 명시할 것.

### GAP-2. 마이그레이션 도구 결정 `supabase db dump` vs 원시 `pg_dump`가 소실됨 — 심각도: 중

- **리서치 근거:** 106행 — 원시 `pg_dump`는 Supabase 내부 스키마/예약 롤까지 포함해 **복원 시 권한 오류**를
  유발하므로, `supabase db dump`(내부 스키마 제외·예약 롤 제거·멱등 `IF NOT EXISTS`)를 써야 한다는
  **명시적 도구 결정**.
- **현황:** addendum은 이관을 "`DATABASE_URL` 환경변수 교체 + 데이터 덤프 수준"으로만 축약했다(20행).
  "어떤 덤프 도구를, 왜"라는 결정의 핵심이 사라졌다.
- **영향:** 이 결정이 빠지면 다운스트림 아키텍처/이관 작업이 원시 `pg_dump`를 선택해 복원 실패를 겪을 수 있다.
  리서치가 콕 집어 경고한 함정이 그대로 재발할 위험.
- **권고:** addendum 이관 항목에 "`supabase db dump` 사용(원시 pg_dump 금지) + 비기본 확장
  (`select * from pg_extension`) 사전 확인" 한 줄을 복원.

### GAP-3. Phase 2 이관 리스크의 완화책(테스트 인스턴스 선검증·점검창)이 누락됨 — 심각도: 중

- **리서치 근거:** 리스크 표 309행 — "Phase 2 이관 데이터 정합성 / 이관 중 데이터 손실 / 완화: 테스트
  인스턴스 선검증, Alembic 스키마 재현 + pg_dump 데이터, **점검창(maintenance window)**".
- **현황:** addendum·PRD 어디에도 이관 검증 절차나 점검창 개념이 없다. PRD NFR6(이식성)은 "코드 변경 없이
  이관 가능"만 말하고 NFR7(데이터 무결성)은 일반 보존만 말한다 — **이관 시 손실 방지 절차**가 비어 있다.
- **권고:** addendum 운영/이관 항목에 "이관은 테스트 인스턴스 선검증 후, 점검창 동안 수행" 절차를 보존하거나
  NFR6/NFR7에 이관 절차 안전성 조건을 1줄 추가.

### GAP-4. 모노레포 학습곡선 리스크와 그 완화책(검증된 스타터 차용)이 소실됨 — 심각도: 하~중

- **리서치 근거:** 리스크 표 307행 — "모노레포 학습곡선 / 초기 셋업 지연 / 완화: 검증된
  스타터(create-t3-turbo) 차용"; 리서치 116·244행에도 `create-t3-turbo`·`turborepo-bun-next-expo`
  스타터가 권고됨.
- **현황:** addendum은 "Turborepo (JS 앱 3개 공유)"만 남기고 **리스크·완화 스타터를 모두 누락**.
- **영향:** 초기 셋업 지연 리스크와 그 구체적 완화 경로(검증 스타터)가 사라져, 다운스트림이 모노레포를
  처음부터 수동 구성하는 비효율을 택할 수 있다.
- **권고:** addendum에 "초기 셋업은 검증된 스타터(create-t3-turbo 등) 차용 권장" 한 줄 추가.

### GAP-5. 자체 인증 보안 리스크의 구체 완화 항목이 일부 축소됨 — 심각도: 하

- **리서치 근거:** 리스크 표 308행 — "자체 인증 보안 취약 / 완화: Argon2 해싱, 짧은 access+refresh,
  시크릿 서버 전용, HTTPS".
- **현황:** PRD NFR3가 해싱·토큰 기반·서버 권한검사·시크릿 비노출·HTTPS를 잘 담았고, addendum 5대 결정 표
  행 3이 Argon2/HS256/토큰 수명을 보존했다 — **대체로 보존됨**. 다만 이것이 "리스크에 대한 의도적 완화책"
  이라는 맥락(리스크-완화 매핑)은 사라졌다. 기능적 손실은 작음.
- **권고:** 필수는 아님. 추적성을 원하면 addendum에 "NFR3 = 자체 인증 보안 리스크 완화"라는 매핑만 부기.

## 충실히 보존되어 검토 통과한 항목 (참고)

- **5대 핵심 기술 결정** 전체 — addendum 표 행 1~5 = 리서치 343~347행 (정확 일치).
- **확정 아키텍처 원칙**(Supabase=DB 전용, 패턴 A, service_role의 RLS 우회 근거) — addendum 14~20행에 보존.
- **기각 대안**(Supabase Auth+RLS / Realtime·WebSocket / BFF) — addendum 55~59행에 근거와 함께 보존.
- **자동 일시정지(2026-02 변경)** — addendum 운영 주의 + PRD NFR8(시연 안정성)에 표면화됨.
- **폴링 채팅 부하 리스크** — PRD NFR5·CM1·NFR9에 반영됨.
- **CORS·Expo Go·토큰 저장·환경변수 함정** — addendum 48~53행에 보존.
- **realtime 200 동시연결 제한 생략** — 폴링 채택으로 무관, **올바른 생략**.
