# PRD Quality Review — gosoom

## Overall verdict

전반적으로 구조와 범위 통제가 뛰어난 PRD다. 거래 루프 하나에 집중하고, FR ID 전역 고유 원칙·범위 제외 목록·해결 항목(R1~R4)을 명시한 점은 다운스트림 친화적이다. 다만 **FR1에 권한 상승(관리자 자가 가입) 모순**과 **FR1→FR20 stale 참조(실제는 FR21)**라는 두 개의 게이트성 결함이 있고, 삭제/비활성화류 FR(FR19·FR22·FR24)의 "완료" 정의가 결과 미명시로 무너진다. **게이트 판정: PASS-WITH-FIXES** — 위 2건은 머지 전 필수 수정.

## 1. 결정 준비성 (Decision-readiness)

**판정: adequate**

핵심 의사결정(폴링 vs 실시간, Supabase=DB 전용, JWT 자체 인증)은 addendum에 근거·기각 대안과 함께 보존되어 다운스트림이 재논의 없이 채택 가능하다. 다만 본문 FR 일부가 "결정을 내린 척"만 하고 운영 결과를 비워둬, 엔지니어가 추가 결정을 강요받는다.

### Findings
- **[high]** 삭제/비활성화 결과 미결정 (§6.6 FR19/FR22/FR24) — "비활성화/삭제 가능"까지만 결정하고 그 *결과*(로그인 차단 여부, 진행 중 채팅·견적 처리, 참조 무결성)는 미결정. 결정 준비성이 아니라 결정 회피. *Fix:* 각 FR에 결과 상태를 명시(예: 비활성 계정은 로그인 거부·신규 견적/메시지 차단, 기존 데이터 유지).
- **[low]** SM 자동 계측 Post-MVP 결정은 명확(R4) — 수동 관찰 시작이 합의됨. 결정 준비성 측면은 양호하나 측정 가능성 문제는 dimension 3 참조. *Fix:* 없음(설계 의도 명확).

## 2. 실속 vs 형식 (Substance over theater)

**판정: strong**

UJ가 페르소나·동작 흐름으로 살아 있고, 반대 지표(CM1~CM3)는 형식적 나열이 아니라 실제 위험(폴링 부하, 견적 스팸, 폼 이탈)을 짚는다. addendum의 "효용 80%를 복잡도 20%로" 같은 판단은 실속 있는 트레이드오프다.

### Findings
- **[medium]** 저작 스캐폴딩 잔존 (§2, line 36 `[NOTE FOR PM]`) — "finished" PRD에 PM용 작업 메모가 남아 있어 미완성 인상. *Fix:* 본문에서 제거하고 내용은 §8 가정/Post-MVP로 흡수.
- **[low]** NFR5/NFR8가 정성적("체감 지연 없이", "안정적으로 기동") — MVP 데모 맥락에선 수용 가능하나 검증 기준이 없다. *Fix:* 데모 합격 기준 한 줄(예: 주요 화면 p95 < 1s, 데모 전 헬스체크 통과) 추가.

## 3. 전략적 일관성 (Strategic coherence)

**판정: adequate**

목표(G1~G3)→지표(SM1~SM4)→범위→FR의 논리 사슬은 일관된다. "데모이면서 실출시 가능한 MVP"라는 포지션도 addendum의 이관 전략(Phase1→2 환경변수 교체)과 정합한다. 다만 지표 일부가 측정 불가라 전략 검증 고리가 끊긴다.

### Findings
- **[high]** SM2·SM3 임계값 부재 (§2, line 27-28) — SM1은 ≥80% 수치가 있으나 SM2("매칭률")·SM3("전환율")은 목표 수치가 없어 달성 판정 불가. 측정 불가 지표는 전략 검증 도구로 무력. *Fix:* SM2/SM3에 데모용 목표치(예: SM2 ≥ 1건/요청, SM3 ≥ 50%) 명시.
- **[low]** CM(반대 지표) 측정 주체 불명 — R4로 자동 계측이 Post-MVP면 CM도 수동 관찰 대상인지 명시 필요. *Fix:* §2 NOTE를 CM에도 적용한다고 한 줄.

## 4. 완료 명확성 (Done-ness clarity) — FR1~FR24

**판정: thin** — 핵심 거래 루프 FR(FR5~FR18)은 양호하나, 관리자·상태전이 FR에서 "완료" 정의가 결과 미명시로 무너진다.

### Findings
- **[critical]** FR1 권한 상승 모순 (§6.1 FR1, line 82) — "역할(고객/고수/**관리자**)을 선택하여 이메일+비밀번호로 회원가입할 수 있다"는 본문이, 같은 줄의 [ASSUMPTION]("관리자 계정은 초기 1개를 시드 보유, 이후 기존 관리자가 추가")과 직접 모순. 문자 그대로 읽으면 누구나 관리자 자가 가입 가능 = 권한 상승. 엔지니어가 "done"을 정의할 수 없을 뿐 아니라 보안 결함을 구현할 위험. *Fix:* FR1 본문을 "고객/고수"만 선택 가능으로 수정하고, 관리자 생성은 FR21 경로로 한정 명시.
- **[high]** FR19 비활성화 결과 미정 (§6.6 FR19, line 110) — "비활성화할 수 있다"의 결과가 없음. 비활성 고객이 로그인/요청 생성 가능한가? 비활성 고수가 채팅 전송·견적 제안 가능한가? 기존 채팅방은? 데이터 유지(R3)만 정해져 있고 *행위 차단*은 공백. 구현 "done" 판정 불가. *Fix:* 비활성 계정의 허용/차단 동작을 열거(로그인 거부 또는 읽기전용, 신규 거래 행위 차단 등).
- **[high]** FR24 사용 중 카테고리 삭제 정책 부재 (§6.6 FR24, line 115) — 관리자가 삭제하는 카테고리를 고수(FR9)·요청(FR5)이 참조 중일 때의 결과 미정(삭제 차단? cascade? soft-delete?). 참조 무결성 공백. *Fix:* "참조 중 카테고리는 삭제 불가 또는 보관(archive) 처리" 규칙 명시.
- **[medium]** FR22 요청 "삭제"의 의미·정합성 (§6.6 FR22, line 113) — "처리(상태 변경/삭제)"의 삭제가 hard delete면 NFR7(데이터 무손실)·FR19 assumption(데이터 유지)과 충돌. soft인지 hard인지 미정. *Fix:* soft-delete(상태 전이) 채택 명시 후 NFR7과 정합.
- **[medium]** FR13/FR12 견적 상태기계 잔류 (§6.4 FR13 line 100 / §6.3 FR12 line 97) — 수락 후 미수락 견적은 "자동 거절하지 않고 정보 보존"이라 영원히 `대기` 상태로 남고, 고수의 견적 목록(FR12)에는 이미 마감(FR10 비활성)된 요청에 `대기`가 계속 표시됨. 고수 입장에서 "done(완료/마감)"이 시각적으로 표현되지 않음 — UJ-2 결말과 어긋남. *Fix:* 수락 시 동형 요청의 잔여 견적을 `마감/만료` 같은 종결 상태로 전이(자동 거절과 구분되는 라벨)하거나, FR12 목록에서 요청 상태를 함께 노출.
- **[low]** FR3 세션 갱신 기준 모호 (§6.1 FR3) — "일정 시간"이 본문엔 없음(addendum에 access 15~30분/refresh 7~30일 존재). 본문만 읽는 다운스트림은 done 기준 불명. *Fix:* 본문에서 addendum 토큰 정책을 1회 참조.

## 5. 범위 정직성 (Scope honesty)

**판정: strong**

제외 목록(§5)이 구체적이고(결제/리뷰/알림/업로드/소셜/검증/실시간), 각 [ASSUMPTION] 태그가 인라인으로 달려 추적 가능하다. 해결 항목 R1~R4가 "구 OQ→FR 반영" 형태로 roundtrip되어 모범적. 출시 지향 MVP 치고 open-item 밀도가 낮은 것은 강점이나, 위험 영역(권한·삭제)이 미해결로 *남았다는 사실 자체*는 미표기다.

### Findings
- **[medium]** 미해결 위험이 Open Questions에 부재 (§8) — FR19/FR22/FR24의 결과 미정, FR1 모순은 실질적 open question인데 §8 "해결된 항목"만 있고 *미해결* 섹션이 비어 정직성에 흠. 출시 스테이크 대비 위험 가시성 부족. *Fix:* §8에 "미해결(Open)" 하위 섹션을 신설해 비활성화 동작·카테고리 삭제 정책·삭제 의미를 등재.
- **[low]** FR23 빈 [ASSUMPTION] (§6.6 FR23, line 114) — 가정 내용 없는 dangling 태그. 가정이 없다면 태그 제거, 있다면(예: 관리자 전 채팅 무차별 열람의 프라이버시 전제) 본문 추가. *Fix:* 태그 제거 또는 가정 채우기.

## 6. 다운스트림 사용성 (Downstream usability)

**판정: adequate** — ID 체계와 도메인 명사는 견고하나, renumber 잔재 stale 참조가 다운스트림 링크를 오도한다.

### Findings
- **[high]** FR1→(FR20) stale 참조 (§6.1 FR1, line 82) — "기존 관리자가 추가**(FR20)**한다"의 FR20은 실제로는 *고수 계정 비활성화*. 관리자 추가는 **FR21**(line 112). mid-draft renumber 잔재로 추정. bmad-create-epics-and-stories/architecture가 잘못된 FR로 링크를 따라감. *Fix:* `(FR20)` → `(FR21)`로 수정.
- **[medium]** UJ/SM 본문 참조 누락 — FR↔UJ 매핑이 명시 ID로 연결되지 않음(예: FR5~FR8이 UJ-1을 구현한다는 역참조 없음). bmad-ux가 여정→화면 매핑 시 수동 추론 필요. *Fix:* 각 UJ 끝에 관련 FR ID 묶음 1줄, 또는 FR 그룹 헤더에 대응 UJ 표기.
- **[low]** addendum 데이터 모델 명사 vs 본문 도메인 명사 정합 양호 — `service_requests/quotes/chat_rooms/messages/categories/users`가 본문 개념(요청/견적/채팅방/메시지/카테고리/계정)과 1:1 대응. 드리프트 없음. *Fix:* 없음(유지).

## 7. 형태 적합성 (Shape fit)

**판정: strong**

소비자 양면 마켓플레이스에 적합한 형태다. UJ-1~UJ-3가 load-bearing(요청→견적→비교→수락→채팅의 양면 교차를 페르소나로 구동)하고, 3역할·3클라이언트 구조가 §3·NFR1·addendum 스택과 일관된다.

### Findings
- **[medium]** 양면 시장 콜드스타트 미언급 (§4 전반) — 마켓플레이스의 핵심 위험인 "한쪽이 없으면 다른 쪽도 무가치"(고수 0명이면 SM2 미달)가 UJ/지표에 반영 안 됨. 데모는 시드 데이터로 회피 가능하나 실출시 지향이면 가정으로라도 표기 필요. *Fix:* §8에 "데모/초기 출시는 고수 시드 계정으로 공급측 부트스트랩" 가정 추가.
- **[low]** UJ-3(관리자)이 거래 루프 밖 운영 여정이라 load-bearing 약함 — 의도된 설계로 보이며 문제 아님. *Fix:* 없음.

## Mechanical notes

- **ID 연속성(FR1~FR24):** 정의는 line 82~115에서 FR1~FR24 모두 정확히 1회씩, 빠짐·중복·건너뜀 없음. 단 **본문 grep의 "FR" 매치 중 line 119~127은 NFR1~NFR9의 부분일치**이며 중복 FR 아님(NFR 번호 정상).
- **stale 참조 1건(critical for downstream):** FR1의 `(FR20)`은 `(FR21)`이어야 함 — renumber 후 미갱신. 그 외 인라인 cross-ref(FR13/FR12/FR9/FR5/FR14/FR10/FR7/FR19)는 *대상까지* 검증 시 모두 정상.
- **가정 인덱스 roundtrip:** 본문 `[ASSUMPTION]` 태그 12개. 그중 **FR23(line 114)은 빈 태그**(가정 텍스트 없음) → 실질 11개. §8 "주요 가정" 요약은 관리자 시드/카테고리/견적·수락/업로드 없음을 커버하나, FR3·FR7·FR9·FR10·FR17·FR23의 개별 가정은 요약에 미반영(인라인에만 존재) — 요약↔인라인 1:1은 아니며 "주요"로 한정한 의도는 명시됨.
- **해결 항목 roundtrip:** R1→FR10, R2→FR7, R3→FR19, R4→§2 NOTE 모두 정상 연결.
- **잔존 스캐폴딩:** §2 line 36 `[NOTE FOR PM]` 제거 필요.
- **글로서리 드리프트:** PRD 도메인 명사 ↔ addendum 엔티티명 드리프트 없음(요청=service_requests 등 일관).
