# 정합성 점검 보고서 (Reconciliation)

원본 입력 `docs/idea.md` 를 완성된 `prd.md` + `addendum.md` 와 대조하여,
입력에서 **누락(DROPPED)** 되었거나 **변형(DISTORTED)** 된 항목 — 특히 기능 요구사항
구조가 조용히 잃어버리기 쉬운 정성적 의도(톤·범위 뉘앙스) — 를 정리한다.

## 총평

전반적으로 **충실도가 높다.** idea.md의 거의 모든 항목이 PRD/Addendum에 명시적으로
표현되었고, 상당수는 오히려 더 상세하게 보강되었다. 핵심으로 점검 요청된
**Supabase→Railway 단계적 이관**은 누락이 아니라 오히려 과(過)보존되었다(아래 확인 항목 참조).
다만 입력의 문자 그대로의 범위와 어긋나는 **미묘한 변형 4건**과, 입력에 없던
**의도/메커니즘 추가**가 존재한다.

---

## 1. 기능 범위 (역할별) 대조

### 고객: 가입 > 로그인 > 서비스 요청 > 견적 확인 > 채팅
- 가입/로그인 → FR1, FR2 ✅
- 서비스 요청 → FR5, FR6, FR7 ✅
- 견적 확인 → FR8 ✅
- 채팅 → FR15~FR18 ✅
- **변형(추가 메커니즘):** 입력의 흐름은 단순히 `견적 확인 > 채팅`인데, PRD는 그 사이에
  **명시적 "견적 수락(FR13)" 게이트**를 채팅 트리거로 삽입했고, **거절(FR14)** 및
  **요청 상태 머신(FR7, FR10)**까지 추가했다. 논리적으로 합당한 연결고리이나 입력에는
  없던 추가 장치다. → **G5 (Low)**

### 고수: 가입 > 로그인 > 카테고리 설정 > 요청 확인 > 견적 제안 > 채팅
- 가입/로그인 → FR1, FR2 ✅
- 카테고리 설정 → FR9 ✅
- 요청 확인 → FR10 ✅
- 견적 제안 → FR11, FR12 ✅
- 채팅 → FR15~FR18 ✅
- 전 항목 충실 반영. 누락 없음.

### 관리자: 가입 > 로그인 > 고객 관리 / 고수 관리 / 관리자 관리 > 요청 관리 > 채팅 관리(내역 확인)
- 로그인 → FR2 ✅
- 고객 관리 → FR19 ✅ / 고수 관리 → FR20 ✅
- 요청 관리 → FR22 ✅
- 채팅 관리(내역 확인) → FR23 (읽기 전용 열람) ✅ — "내역 확인"이라는 입력 뉘앙스와 정확히 일치
- 카테고리 관리 → FR24 ✅ (입력에 명시 안 됐으나 합리적 보강)
- **변형 1 (헤드라인):** 입력은 관리자도 `가입 > 로그인`을 명시했으나, PRD FR1의
  [ASSUMPTION]은 관리자 계정을 **시스템 시드 + 기존 관리자가 추가(FR21)**하는 방식으로
  제한해 자가 가입을 사실상 막았다. 가정으로 문서화되어 완전히 조용한 변경은 아니지만,
  입력의 문자적 의미와는 어긋난다. → **G1 (Medium)**
- **변형 2 (범위 축소):** 입력의 `관리자 관리`는 관리자를 *관리(manage)*한다는 의미인데,
  FR21은 **조회 + 추가만** 허용하고 비활성화/수정이 없다(고객·고수에는 FR19/FR20으로
  비활성화 허용 — 대비됨). → **G2 (Low)**

## 2. 기술 스택 대조

| idea.md 항목 | PRD/Addendum 반영 | 상태 |
|---|---|---|
| 1. 사용자(고객/고수)용 반응형 웹(Next.js) | NFR1, Addendum 기술스택표 (Next.js App Router) | ✅ |
| 2. 관리자용 반응형 웹(Next.js) | NFR1, Addendum 기술스택표 | ✅ |
| 3. 사용자용 모바일 앱(React Native) | NFR1, Addendum (RN + Expo SDK 52+) | ✅ |
| 4. 통합 API(FastAPI) | NFR4, Addendum (router→service→repository) | ✅ |
| 5. DB (Supabase, PostgreSQL) | Addendum (Supabase=DB 전용, SQLAlchemy+Alembic) | ✅ |
| 6. 배포(Railway: FE/BE/PG), 테스트(expo go) | NFR1(Expo Go), Addendum 배포행 | ✅ |

기술 스택은 전부 반영됨. 누락 없음.

**입력에 없던 추가(=드롭 아님, 보강):** SQLAlchemy/Alembic, Turborepo 모노레포,
Argon2/JWT(HS256, access 15~30분 + refresh 7~30일) 세부, Cloudflare R2(Post-MVP),
채팅 폴링 2~3초 구체화. 모두 Addendum에 기술 결정으로 보존. → **G6 (Low, 추가 항목)**

## 3. Supabase → Railway 단계적 이관 (요청된 핵심 점검 항목)

**결론: 누락 없음. 오히려 충실히/과보존됨.**

- 입력의 "[1단계: Supabase 개발] → [2단계: Railway PostgreSQL 독립 이관]" 흐름 →
  **NFR6**(기능 코드 변경 없이 이관 가능) + **Addendum "확정된 아키텍처 원칙"**으로 명확히 표현.
- Addendum이 입력보다 더 구체화: 이관 = **`DATABASE_URL` 환경변수 교체 + 데이터 덤프 수준,
  앱 코드 변경 없음**. "Supabase는 DB(PostgreSQL)로만 사용, 인증/실시간/스토리지는 FastAPI 소유"
  원칙으로 *왜* 이관이 쉬운지까지 보존.
- 기각 대안(Supabase Auth + RLS → 이관 시 전원 재인증 필요)으로 결정 근거까지 남김.
- 운영 주의(Supabase 무료 티어 1주 비활성 시 자동 일시정지 → 시연 직전 깨우기)는 NFR8과 연결.

이 항목은 정성적 의도가 잘 보존된 모범 사례.

---

## 4. 정성적 톤/의도 (FR 구조가 잃기 쉬운 부분)

- **변형 3 (범위 인플레이션):** idea.md의 톤은 단호하게 **최소/데모 지향**이다
  ("최소 기능만 구현", "시연에 사용하는 기술 스택"). 그러나 PRD 개요와 **G3**은
  "**실제 출시까지 이어질 수 있는 견고한 MVP**", "실제 출시로 확장 가능한 구조"를 추가했다.
  합리적 방향이나 입력이 명시하지 않은 의도가 주입됐다 — 본 점검이 주목하는 "FR 구조가
  옮겨버리는 정성적 의도"의 전형. → **G3 (Low-Medium)**
- "숨고와 유사" 라는 입력의 정체성은 개요에 잘 반영됨 ✅.

---

## 5. 내부 일관성 메모

- **FR1 자기모순:** FR1은 관리자(고객/고수/관리자)를 **가입 시 선택 가능한 역할**로 나열하면서도,
  같은 줄의 [ASSUMPTION]은 관리자 자가 가입을 부정(시드 + 기존 관리자 추가)한다. 한 문장 안에서
  서로 충돌. 사용자 확인 필요. → **G4 (Low)**

---

## 갭 요약 (심각도순)

| ID | 유형 | 내용 | 심각도 |
|----|------|------|--------|
| **G1** | 변형 | 관리자 자가 가입(입력 명시) → 시드+관리자 추가 방식으로 제한 (FR1 ASSUMPTION) | **Medium** |
| **G3** | 변형(톤) | 입력의 "최소/데모" 의도 → "실제 출시 가능한 견고한 MVP"로 범위 확장 (개요, G3) | Low-Medium |
| **G2** | 변형(축소) | `관리자 관리`(manage) → 조회+추가만, 비활성화/수정 없음 (FR21) | Low |
| **G4** | 일관성 | FR1이 관리자를 가입 선택 역할로 나열 ↔ 같은 줄 ASSUMPTION이 부정 | Low |
| **G5** | 추가 | 입력 `견적확인>채팅` 사이에 수락 게이트(FR13)+거절(FR14)+상태머신 삽입 | Low |
| **G6** | 추가 | SQLAlchemy/Alembic·Turborepo·Argon2/JWT·R2·폴링 2~3초 등 입력에 없던 기술 결정 | Low |

> **드롭(완전 누락) 0건.** 모든 갭은 변형/축소/추가 또는 내부 일관성 이슈이며,
> 가장 주의할 것은 **G1(관리자 자가 가입 제한)** — 사용자(KTH) 확인을 권장한다.
